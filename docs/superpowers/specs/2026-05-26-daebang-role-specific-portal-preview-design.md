# 대방동지역주택조합 역할별 포털 미리보기 설계 명세

작성일: 2026-05-26
상태: 구현 계획 전 사용자 검토 대상

## 1. 문서 목적

이 문서는 기존 `2026-05-25-daebang-housing-cooperative-portal-design.md`의 1단계 범위 중 아직 구현되지 않은 **로그인 이후 역할별 대시보드 화면 구조와 빈 상태**를 상세화한다.

이번 구현은 공개 랜딩페이지에서 안내된 조합원 전용 서비스를 역할별 개별 페이지로 미리 보여주는 UI 작업이다. 실제 인증, 권한 판정, 문서 게시·열람·다운로드, 회계 데이터 처리, 알림 발송, 투표 참여는 포함하지 않는다.

## 2. 유지되는 기존 결정

이 문서는 다음 기존 결정을 변경하지 않는다.

- 공개 내비게이션은 `조합소개`, `사업정보`, `조합소식`, `자료실`, `로그인`으로 유지한다.
- `정보공개`, `회계·실적보고`, `내 분담금`, `내 환불현황`, `이슈의 장`, `투표·설문`은 로그인 이후 영역으로만 표현한다.
- 일반 방문자에게 내부 문서, 개인별 자료, 회계 원문 또는 조합원 전용 서식을 공개하는 페이지를 새로 만들지 않는다.
- 현재 `/login`은 인증 기능이 아닌 준비 안내 진입점이며, 역할별 화면 역시 서비스 구조를 설명하는 미리보기다.
- 첫 구현 단계는 프런트엔드 정적 UI와 빈 상태에 한정한다.

## 3. 선택한 접근

### 선택: 공통 포털 셸과 역할별 정적 콘텐츠를 사용하는 개별 라우트

역할마다 주소가 분리된 페이지를 제공하되, 화면 골격과 안내 표현은 공통 컴포넌트로 구성한다.

| 라우트 | 대상 역할 | 목적 |
|---|---|---|
| `/portal/member` | 정식 조합원 | 조합원 전용 공개자료, 회계보고, 분담금, 참여 기능의 이용 구조 안내 |
| `/portal/refund` | 환불조합원 | 허용된 자료와 본인 환불현황 중심의 이용 구조 안내 |
| `/portal/admin` | 관리자 | 향후 검토·승인·운영관리 화면의 구조 안내 |

이 방식은 개별 페이지를 원하는 사용자 방향을 충족하면서도, 공통 안내와 카드 표현을 한곳에서 관리할 수 있다. 추후 인증과 역할 권한이 도입되면 각 라우트의 접근 제어를 추가하고 기존 화면 컴포넌트를 재사용할 수 있다.

### 채택하지 않는 방식

- 역할별 페이지에 전체 마크업을 각각 작성하는 방식은 초기 작성은 빠르지만 안내 문구와 카드 상태 표현이 중복된다.
- `/portal?role=member`와 같은 단일 쿼리 페이지 방식은 역할별 진입점과 향후 접근 제어 경계를 분명히 보여주지 못한다.

## 4. 사용자 흐름과 라우트

### 4.1 공개 영역

공개 랜딩페이지는 기존과 동일하게 조합원 전용 서비스가 로그인 후 이용 가능하다는 안내만 제공한다. 공개 헤더나 공개 바로가기 카드에 역할별 포털 페이지를 직접 노출하지 않는다.

### 4.2 로그인 안내 페이지

`/login`은 실제 로그인 폼을 추가하지 않는다. 현재의 준비 안내를 유지하면서, 별도의 `포털 화면 미리보기` 구역에 아래 세 진입 링크를 제공한다.

- `정식 조합원 화면 보기` -> `/portal/member`
- `환불조합원 화면 보기` -> `/portal/refund`
- `관리자 화면 보기` -> `/portal/admin`

세 링크에는 `미리보기` 또는 `준비 화면`임을 명확히 표시해 인증 없이 실제 서비스에 들어간다는 오해를 방지한다.

### 4.3 포털 페이지 공통 이동

각 역할별 페이지는 최소한 아래 이동 수단을 가진다.

- 랜딩페이지로 돌아가기
- 로그인 안내 페이지로 돌아가기
- 같은 미리보기 범위 내에서 다른 역할 화면을 확인할 수 있는 역할 전환 링크

역할 전환은 데모 확인을 위한 탐색 기능이며 권한 변경이나 로그인 상태를 의미하지 않는다.

## 5. 공통 화면 구조

각 역할별 포털 페이지는 다음 순서로 구성한다.

1. **간결한 상단 내비게이션**
   - 대방동지역주택조합 명칭 또는 로고 표현
   - `홈으로`, `로그인 안내` 이동
   - 현재 화면에 `미리보기` 배지 표시
2. **역할 헤더**
   - 역할명
   - 해당 역할이 향후 확인하게 될 정보의 목적 설명
   - 역할별 포인트 카드 또는 요약 상태
3. **운영 준비 안내 배너**
   - 이 페이지는 화면 구조 미리보기라는 문구
   - 실제 인증, 개인정보, 자료 열람, 처리 기능을 제공하지 않는다는 문구
4. **우선 서비스 카드 그리드**
   - 역할별 허용 메뉴만 노출
   - 카드마다 `준비 중`, `로그인 후 제공 예정`, `권한 확인 후 제공 예정`과 같은 상태를 표시
   - 실행 가능한 열람·다운로드·승인·발송 버튼은 제공하지 않음
5. **빈 상태 또는 이용 안내 패널**
   - 아직 표시 가능한 문서, 알림 또는 개인별 데이터가 없다는 상태
   - 실제 데이터 연동 이후 제공될 내용을 과장 없이 설명

## 6. 역할별 콘텐츠

### 6.1 정식 조합원 `/portal/member`

페이지 제목: `정식 조합원 포털 미리보기`

안내 목적: 정식 조합원이 향후 본인 정보와 조합 운영 정보를 한곳에서 확인하고 참여할 수 있음을 보여준다.

| 카드 | 설명 | 표시 상태 |
|---|---|---|
| 확인 필요 알림 | 중요공지와 확인 요청이 표시될 영역 | `알림 연동 준비 중` |
| 내 분담금 | 본인의 납부·미납 현황이 표시될 영역 | `개인별 자료 제공 예정` |
| 새 정보공개 | 권한이 있는 공개자료가 표시될 영역 | `게시 기능 준비 중` |
| 회계·실적보고 | 허용된 예산·결산 및 추진실적을 확인할 영역 | `열람 기능 준비 중` |
| 이슈의 장 | 현안 토론과 공식답변이 표시될 영역 | `참여 기능 준비 중` |
| 투표·설문 | 의견수렴과 투표가 표시될 영역 | `투표 기능 준비 중` |

금지 표현:

- 실제 납부액, 미납액 또는 납부기한을 예시 데이터로 표시하지 않는다.
- 실제 문서 건수나 새 알림이 존재하는 것처럼 숫자 배지를 표시하지 않는다.

### 6.2 환불조합원 `/portal/refund`

페이지 제목: `환불조합원 포털 미리보기`

안내 목적: 환불 완료 전 허용 범위의 정보와 본인 처리현황을 확인하는 흐름을 보여준다.

| 카드 | 설명 | 표시 상태 |
|---|---|---|
| 공개자료 | 환불조합원에게 허용된 공개자료 영역 | `권한 확인 후 제공 예정` |
| 회계·실적보고 | 허용 범위의 회계·실적자료 영역 | `제공 범위 확정 예정` |
| 내 환불현황 | 본인의 정산 통지와 처리현황 영역 | `개인별 자료 제공 예정` |
| 통지 알림 | 본인 대상 통지와 확인 안내 영역 | `알림 연동 준비 중` |

표시하지 않는 기능:

- `내 분담금`
- `이슈의 장`
- `투표·설문`
- 관리자 승인·발송·권한관리 기능

### 6.3 관리자 `/portal/admin`

페이지 제목: `관리자 포털 미리보기`

안내 목적: 운영자가 향후 검토와 승인을 수행하게 될 관리 화면의 우선순위를 보여준다.

| 카드 | 설명 | 표시 상태 |
|---|---|---|
| 공개기한 | 정보공개 대상과 마감일 점검 영역 | `관리 기능 준비 중` |
| 문서 승인 | 공개 또는 회계자료의 검토 대기 영역 | `승인 절차 준비 중` |
| 납부자료 검증 | 향후 엑셀 업로드 자료 확인 영역 | `업로드 연동 준비 중` |
| 독촉 승인 | 발송 전 검토와 승인 영역 | `발송 연동 준비 중` |
| 알림 결과 | 발송 결과 및 오류 확인 영역 | `결과 조회 준비 중` |
| 권한·감사로그 | 역할 변경과 기록 확인 영역 | `감사 기능 준비 중` |

관리자 화면에서도 실제 승인, 업로드, 발송 또는 권한 변경을 수행하는 활성 버튼은 제공하지 않는다.

## 7. 시각 및 인터랙션 기준

포털 미리보기는 기존 랜딩페이지와 같은 디자인 시스템을 사용한다.

- 배경은 warm canvas `#fbfaf9`를 유지한다.
- 카드에는 흰 면과 stone inset outline을 사용하고, 상태 안내 영역은 recessed panel `#f8f7f4`로 구분한다.
- 주 이동 CTA가 필요한 경우 dark pill 버튼을 사용하되, 작동하지 않는 서비스 기능을 활성 버튼처럼 표현하지 않는다.
- 모든 텍스트는 Pretendard를 사용한다.
- 서비스 종류별 구분은 작은 색상 포인트와 텍스트 상태값으로 처리하며, 신규 이미지 자산은 이번 범위에 필요하지 않다.
- 애니메이션을 추가할 경우 장식 수준으로 제한하며 `prefers-reduced-motion: reduce`에서 정지한다.
- 데스크톱에서는 카드 그리드, 모바일에서는 한 열의 읽기 쉬운 목록으로 표시한다.

## 8. 컴포넌트와 데이터 구조

### 8.1 파일 구조

| 파일 | 책임 |

| `src/components/portal/portal-shell.tsx` | 공통 상단 내비게이션, 역할 헤더, 미리보기 안내 배너, 기본 레이아웃 |

### 8.2 정적 데이터 계약

콘텐츠는 구현 컴포넌트 내부에 흩어 쓰지 않고 타입이 있는 정적 데이터로 둔다. 최소 데이터 구조는 다음 의미를 가진다.

- `PortalRole`: `member`, `refund`, `admin`
- 역할별 `href`, `title`, `description`
- 역할별 카드 배열: `title`, `description`, `status`, 선택적 색상 키
- 역할별 빈 상태 제목과 안내 문구

라우트 페이지는 역할 키를 선택해 공통 화면에 전달하는 얇은 진입점으로 유지한다. 현재는 정적 데이터만 사용하고 API, 세션, 쿠키, 데이터베이스 또는 파일 다운로드를 연결하지 않는다.

## 9. 접근성, 상태 표현 및 오류 방지

- 페이지 최상위에는 역할명이 포함된 하나의 명확한 `h1`을 제공한다.
- 상태값은 색상만으로 구분하지 않고 텍스트로 표시한다.
- 이동 링크는 키보드로 접근 가능하고 현재 미리보기 역할을 식별할 수 있어야 한다.
- 미구현 기능 카드는 링크나 버튼으로 위장하지 않으며, 클릭해도 동작할 것처럼 보이는 hover·cursor 표현을 쓰지 않는다.
- 존재하지 않는 개인자료, 문서, 알림을 실제 데이터처럼 보여주는 가짜 성공 상태는 만들지 않는다.
- 잘못된 역할 URL을 별도로 지원하지 않으며, 정의된 세 개의 정적 라우트만 생성한다.

## 10. 테스트와 검증 기준

### 컴포넌트 테스트

`src/__tests__/portal-preview-pages.test.tsx`는 다음을 확인한다.

- `/login` 내용에 세 역할별 미리보기 링크와 실제 인증 기능이 준비 중이라는 안내가 있다.
- 정식 조합원 페이지에는 `내 분담금`, `정보공개`, `이슈의 장`, `투표·설문`이 표시된다.
- 환불조합원 페이지에는 `내 환불현황`이 표시되고 `내 분담금`, `이슈의 장`, `투표·설문`은 표시되지 않는다.
- 관리자 페이지에는 `문서 승인`, `독촉 승인`, `권한·감사로그`가 표시된다.
- 모든 역할별 페이지에 `미리보기` 및 실제 인증·개인자료 제공 기능이 아니라는 안내가 표시된다.
- 각 페이지에서 홈과 로그인 안내 페이지로 이동할 수 있다.

### 브라우저 확인

보이는 화면 변경이므로 실행 중인 페이지를 Codex 브라우저에서 확인한다.

- 데스크톱: 세 역할별 카드 구분, 안내 배너, 페이지 간 이동, 랜딩/로그인 복귀 경로 확인
- 모바일: 카드가 한 열로 읽히고, 가로 넘침이 없으며, 역할 전환 링크와 안내 문구가 잘리지 않는지 확인
- 접근성 상태: 키보드 포커스와 reduced-motion 설정에서 장식 모션이 방해되지 않는지 확인

## 11. 이번 구현에서 하지 않는 것

다음 작업은 별도 설계와 구현 계획 없이는 진행하지 않는다.

- 로그인 폼, 인증 제공자, 회원가입 또는 세션 저장
- 실제 역할 판정, 권한 가드, 역할 변경
- 정보공개 문서 업로드·다운로드·검색·게시
- 회계·실적보고 데이터 조회 또는 엑셀 업로드
- 개인별 분담금·환불현황 데이터 표시
- 알림톡, SMS, 이메일 또는 발송 승인 처리
- 댓글, 투표, 설문 또는 관리자 승인 동작
- 공개 메뉴에 조합원 전용 자료를 추가하는 변경

## 12. 후속 구현 계획 경계

이 명세가 승인되면 다음 구현 계획은 정적 미리보기 화면만 대상으로 작성한다.

1. 역할별 콘텐츠 계약과 실패 테스트 작성
2. 공통 포털 셸 및 카드/빈 상태 UI 구현
3. 세 개의 역할별 정적 라우트 연결
4. `/login`에 오해 없는 미리보기 진입 경로 추가
5. 테스트, 빌드, 데스크톱·모바일 화면 검증

인증·권한·실데이터 연동은 기존 상위 명세의 2단계 이후 별도 명세에서 다룬다.
