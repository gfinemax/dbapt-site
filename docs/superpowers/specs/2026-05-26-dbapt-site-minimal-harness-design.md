# dbapt-site 최소 하네스 설계 명세

작성일: 2026-05-26
상태: 하네스 산출물 구현 계획 전 사용자 검토 대상

## 1. 목적

`dbapt-site`는 공개 랜딩페이지 이후 역할별 포털 미리보기, 인증·권한, 문서공개, 회계·알림 등 순차적인 기능 확장이 예정된 프로젝트다. 이 문서는 각 기능의 제품 요구사항을 새로 정의하지 않고, 승인된 스펙을 구현하고 검증하는 과정이 반복 가능하도록 저장소 내부에 추가할 최소 하네스 구조를 정한다.

하네스의 역할은 다음과 같다.

- 제품 스펙과 구현 계획 사이의 연결 규칙을 명확히 한다.
- 공개 영역과 로그인 이후 영역의 경계가 구현 중 흐려지지 않도록 검토 단계를 둔다.
- UI 구현 결과의 품질 확인과 검증 기록을 같은 형식으로 남긴다.
- 기능 범위가 커질 때 필요한 전문 스킬만 추가할 수 있는 출발점을 만든다.

## 2. 기준과 범위

### 기준 문서

하네스는 아래 기존 문서를 입력으로 사용한다.

| 문서 | 역할 |
|---|---|
| `AGENTS.md` | 항상 지켜야 하는 기술·디자인·검증 규칙 |
| `DESIGN.md` | UI 시각 기준과 토큰 |
| `docs/superpowers/specs/2026-05-25-daebang-housing-cooperative-portal-design.md` | 제품 전체 단계와 공개/로그인 경계 |
| `docs/superpowers/specs/2026-05-26-daebang-role-specific-portal-preview-design.md` | 다음 UI 구현 대상인 역할별 포털 미리보기 상세 설계 |
| `docs/superpowers/plans/*.md` | 승인된 설계를 실행 단위로 풀어낸 구현 계획 |

### 이번 하네스에서 다루는 것

- 기능 스펙을 입력으로 받아 구현 계획, 구현, 검토, 검증을 순차적으로 진행하는 규약
- UI 구현 범위와 표현의 정직성을 점검하는 재사용 검토 스킬
- 단계별 기록을 남기는 `_workspace/` 경로 규칙
- `AGENTS.md`에서 하네스 문서를 찾을 수 있도록 하는 짧은 연결 지침

### 이번 하네스에서 다루지 않는 것

- 제품 기능 요구사항 변경 또는 역할 권한 정책 변경
- 실제 로그인, 문서공개, 회계, 발송, 투표 구현
- 인증·문서·회계 같은 미래 기능 전용 전문 스킬 선행 생성
- 병렬 다중 에이전트 운영이나 동적 작업 배분
- 모델별 복구 프롬프트, 재시도 요령 또는 특정 런타임 종속 로직

## 3. 선택한 하네스 구조

### 외부 패턴: Pipeline

현재 작업은 승인된 설계가 먼저 존재하고, 이후 계획과 구현, 검토 및 검증이 순서대로 진행되어야 한다. 따라서 하네스의 바깥 구조는 순차 의존 작업에 맞는 `Pipeline`으로 둔다.

### 품질 게이트: 제한된 Producer-Reviewer

모든 작업을 복잡한 리뷰 체계로 만들지는 않는다. 보이는 UI 변경 또는 공개/로그인 기능 경계에 영향을 주는 변경에만 `dbapt-site-ui-review` 검토를 의무화한다. 이 검토는 새 기능을 설계하거나 재구현하는 역할이 아니라, 이미 승인된 범위에 맞는지 판정하는 품질 게이트다.

### 채택하지 않는 구조

| 구조 | 채택하지 않는 이유 |
|---|---|
| 단일 실행 스킬만 두는 방식 | 범위 위반과 UI 과장 표현을 독립적으로 확인할 검토 기록이 남지 않는다. |
| Fan-out/Fan-in | 현재 역할별 포털 미리보기는 병렬 전문 작업보다 순차 구현과 일관성 검토가 중요하다. |
| Supervisor 또는 Hierarchical Delegation | 동적으로 변하는 작업 큐나 복수 하위 도메인 조정이 아직 필요하지 않다. |

## 4. 산출물 구조

하네스 구현 시 생성할 최소 파일은 다음과 같다.

```text
dbapt-site/
├── AGENTS.md
├── .agents/
│   └── skills/
│       ├── dbapt-site-orchestrator/
│       │   └── SKILL.md
│       └── dbapt-site-ui-review/
│           └── SKILL.md
├── docs/
│   └── harness/
│       └── dbapt-site/
│           └── team-spec.md
└── _workspace/
    └── README.md
```

### 파일별 책임

| 파일 | 책임 |
|---|---|
| `AGENTS.md` | 기존 상시 규칙을 유지하고, 반복 작업 흐름은 `docs/harness/dbapt-site/team-spec.md`를 참조하도록 포인터 추가 |
| `.agents/skills/dbapt-site-orchestrator/SKILL.md` | 승인된 스펙을 받아 계획 작성, 구현, UI 검토, 검증 기록으로 이어가는 순차 흐름 정의 |
| `.agents/skills/dbapt-site-ui-review/SKILL.md` | 공개/로그인 경계, 미구현 기능의 과장 표현, 디자인 및 접근성 규칙 준수 여부 검토 |
| `docs/harness/dbapt-site/team-spec.md` | 목표, 역할, 단계, 인수인계 산출물, 실패 정책, 검증 기준을 지속 가능한 문서로 관리 |
| `_workspace/README.md` | 구현 작업마다 생성될 임시 기록 파일의 명명 규칙과 보존 범위 정의 |

`_workspace/`에는 하네스 설치 시 안내 문서만 만들고, 실제 요청 요약·검토·검증 기록은 다음 기능 구현이 시작될 때 생성한다. 하네스 자체를 세팅하는 단계에서 존재하지 않는 구현 결과를 미리 기록하지 않는다.

## 5. 역할 정의

### 5.1 `dbapt-site-orchestrator`

목적: 승인된 제품 스펙 하나를 안전한 구현 결과까지 연결한다.

입력:

- 사용자의 구현 요청
- 해당 기능의 승인된 `docs/superpowers/specs/*.md`
- 승인된 또는 새로 작성할 `docs/superpowers/plans/*.md`
- `AGENTS.md`와, UI 변경 시 `DESIGN.md`

책임:

1. 어떤 스펙과 계획이 현재 요청의 기준인지 확인한다.
2. 계획이 없으면 Superpowers 계획 작성 흐름으로 연결한다.
3. 계획이 승인된 뒤에만 코드 구현을 시작한다.
4. UI 변경 또는 접근 범위 변경이면 `dbapt-site-ui-review` 결과를 요구한다.
5. 저장소 규칙에 맞는 검증 결과가 확보된 뒤 완료를 보고한다.

하지 않는 일:

- 제품 요구사항을 임의로 확대하지 않는다.
- 승인받지 않은 인증·데이터·외부 연동 기능을 구현하지 않는다.
- 불확실한 접근 권한이나 노출 범위를 혼자 결정하지 않는다.

### 5.2 `dbapt-site-ui-review`

목적: 구현 화면이 승인된 범위와 프로젝트 디자인 규칙에 맞는지 검토한다.

입력:

- 원래 사용자 요청과 승인 스펙
- 구현 계획
- 변경된 UI 코드 또는 렌더링 결과
- `AGENTS.md`, UI 관련 변경이면 `DESIGN.md`

검토 항목:

- 공개 내비게이션과 공개 페이지에 로그인 전용 기능이 직접 노출되지 않는가
- 실제 인증, 문서 열람, 개인 데이터, 승인 또는 발송 기능이 제공되는 것처럼 표현되지 않는가
- 역할별 화면의 포함·제외 항목이 승인된 스펙에 맞는가
- Pretendard, 지정 색상·자산·카드 스타일·모션 제한을 준수하는가
- 데스크톱과 모바일에서 읽기 흐름, 가로 넘침, 포커스 접근성이 적절한가

출력 판정:

- `PASS`: 범위 위반이나 사용자에게 오해를 줄 표현이 없음
- `FIX`: 구체적으로 수정 가능한 위반이 있음
- `ESCALATE`: 스펙 또는 운영 정책 결정이 없이는 판단할 수 없음

검토자는 새로운 기능을 요구하거나 디자인을 다시 창작하지 않고, 승인된 기준에 대한 증거 기반 판정만 남긴다.

## 6. 단계 흐름과 인수인계 파일

실제 기능 구현 작업은 다음 파이프라인을 따른다.

| 단계 | 책임 | 입력 | 출력 |
|---|---|---|---|
| 0. 요청 정리 | orchestrator | 사용자 요청, 현재 저장소 상태 | `_workspace/00_input/request-summary.md` |
| 1. 기준 확정 | orchestrator | 요청 요약, 제품 스펙 | `_workspace/01_scope/spec-selection.md` |
| 2. 구현 계획 | Superpowers 계획 흐름 | 승인 스펙 | `docs/superpowers/plans/YYYY-MM-DD-<topic>.md` |
| 3. 구현 | 구현 담당 | 승인된 계획 | 코드·테스트 변경 |
| 4. UI 범위 검토 | ui-review | 요청, 스펙, 계획, 변경 결과 | `_workspace/04_review/ui-review.md` |
| 5. 검증 및 완료 | orchestrator | 변경 결과, 검토 결과 | `_workspace/final/verification.md` |

### 입력 요약 파일

`_workspace/00_input/request-summary.md`는 해당 구현 회차의 기준을 분명히 하기 위해 아래만 기록한다.

- 사용자 요청
- 현재 대상으로 선택된 기능
- 참조해야 할 승인 스펙
- 명시적으로 제외된 범위
- 확인이 필요한 결정이 있는지 여부

### UI 검토 파일

`_workspace/04_review/ui-review.md`는 아래 구조를 가진다.

- 검토 대상 변경
- 읽은 기준 문서
- 공개/로그인 범위 판정
- 기능 표현 정직성 판정
- 디자인·반응형·접근성 판정
- 최종 결과: `PASS`, `FIX`, `ESCALATE`

### 최종 검증 파일

`_workspace/final/verification.md`에는 실제 실행한 확인만 기록한다.

- 구현 대상과 관련 커밋 또는 변경 파일
- `pnpm lint`, `pnpm test`, `pnpm build` 결과
- 보이는 변경일 경우 데스크톱·모바일 브라우저 확인 결과
- 해결되지 않은 위험 또는 후속 스펙 필요 항목

## 7. Superpowers 문서 흐름과의 관계

하네스는 현재 사용 중인 Superpowers 설계·구현 문서를 대체하지 않는다.

| 상황 | 기준 흐름 |
|---|---|
| 새 기능의 범위 또는 UI 구조를 결정할 때 | `docs/superpowers/specs/`에 설계 명세 작성 및 승인 |
| 승인된 기능을 구현 단위로 분해할 때 | `docs/superpowers/plans/`에 구현 계획 작성 |
| 계획에 따라 실제 작업을 관리할 때 | 하네스 orchestrator와 `_workspace/` 기록 사용 |
| 구현이 승인 범위와 화면 규칙을 지켰는지 확인할 때 | `dbapt-site-ui-review` 사용 |

따라서 현재 작성된 역할별 포털 미리보기 스펙은 폐기하거나 옮기지 않는다. 하네스 적용 후 첫 실제 사용 대상은 해당 스펙의 구현 계획과 UI 구현 작업이다.

## 8. `AGENTS.md` 연결 원칙

현재 `AGENTS.md`는 UI 구현에 필요한 안정적인 상시 규칙을 이미 담고 있다. 하네스 구현 단계에서는 내용을 대체하거나 긴 절차를 복사하지 않고, 다음 의미의 짧은 항목만 추가한다.

- 승인된 기능을 계획·구현·검증하는 반복 흐름은 `docs/harness/dbapt-site/team-spec.md`를 따른다.
- UI 또는 공개/로그인 영역 변경은 해당 팀 스펙의 UI 검토 단계를 통과해야 한다.

이렇게 하면 `AGENTS.md`는 모든 세션에 필요한 핵심 규칙만 유지하고, 단계별 세부 절차는 하네스 문서가 맡는다.

## 9. 실패 및 중단 정책

| 상황 | 처리 |
|---|---|
| 승인된 스펙이 없음 | 구현하지 않고 설계 승인 단계로 되돌린다. |
| 요청과 기존 스펙이 충돌함 | `_workspace/01_scope/spec-selection.md`에 충돌을 기록하고 사용자 확인을 받는다. |
| UI 검토가 `FIX`임 | 수정 후 한 번 더 검토하며, 통과 전 완료로 보고하지 않는다. |
| UI 검토가 `ESCALATE`임 | 임의 구현하지 않고 필요한 정책 결정을 질문한다. |
| 검증 명령 실패 | 실패 결과를 남기고 원인을 수정하거나, 해결되지 않으면 완료로 보고하지 않는다. |
| 브라우저 검증을 수행할 수 없음 | 수행하지 못한 이유와 남은 위험을 검증 기록에 명시한다. |

수정-검토 반복은 기본적으로 한 차례 수정 후 재검토까지로 제한한다. 추가 반복이 필요할 정도로 범위 문제가 커지면 구현 수정보다 스펙 보완이 필요한지 먼저 판단한다.

## 10. 검증 기준

### 구조 검증

- `.agents/skills/dbapt-site-orchestrator/SKILL.md`와 `.agents/skills/dbapt-site-ui-review/SKILL.md`가 YAML frontmatter의 `name`, `description`을 가진다.
- `docs/harness/dbapt-site/team-spec.md`와 각 스킬에서 사용하는 역할명과 `_workspace/` 경로가 일치한다.
- `AGENTS.md`의 하네스 포인터가 실제 팀 스펙 파일을 가리킨다.
- `_workspace/README.md`는 임시 작업 기록 경로와 보존 원칙을 설명한다.

### 시나리오 검증

하네스 구현 후 아래 두 시나리오를 문서 수준으로 점검한다.

1. 정상 흐름: 승인된 역할별 포털 미리보기 스펙을 받아 구현 계획과 UI 검증 기록까지 연결할 수 있다.
2. 실패 흐름: 공개 메뉴에 로그인 전용 자료를 추가하려는 구현이 검토 단계에서 `FIX` 또는 `ESCALATE`로 차단된다.

### 제품 구현 검증

하네스가 적용되는 실제 UI 변경은 기존 저장소 규칙에 따라 다음 검증을 수행한다.

```powershell
pnpm lint
pnpm test
pnpm build
```

보이는 변경이면 Codex 브라우저에서 데스크톱과 모바일 레이아웃도 확인한다.

## 11. 구현 순서

이 설계가 승인되면 별도의 구현 계획은 다음 순서로 하네스 산출물을 만든다.

1. `docs/harness/dbapt-site/team-spec.md` 작성
2. `.agents/skills/dbapt-site-orchestrator/SKILL.md` 작성
3. `.agents/skills/dbapt-site-ui-review/SKILL.md` 작성
4. `_workspace/README.md` 작성
5. `AGENTS.md`에 짧은 하네스 문서 포인터 추가
6. 경로, frontmatter, 정상·실패 시나리오 정합성 검증

하네스 작성 완료 후에야 역할별 포털 미리보기 구현 계획으로 돌아간다.
